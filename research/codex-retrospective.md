# Codex Retrospective

Локальный протокол дополняет общий skill `C:\Users\HONOR\.codex\skills\codex-retrospective\SKILL.md`. Если они расходятся, сначала сверяйся с общим skill и правь этот файл коротко, без дублирования всего текста.

## Когда применять

- В конце длинной или существенной сессии.
- Когда пользователь просит улучшить процесс, правила, skill или рабочие заметки.
- Когда пользователь пишет кодовое слово `Ретроспектива` или `ретроспектива`.

## Протокол

1. Перечитай общий `codex-retrospective` skill и этот файл.
2. Перечитай `AGENTS.md`, если вывод может быть проектным правилом.
3. Найди повторяемую ошибку или устойчивое предпочтение; не ограничивайся пересказом сессии.
4. Выбери один правильный слой для правки:
   - существующий skill — повторяемый workflow с понятным триггером;
   - новый skill — новый повторяемый workflow, который не покрыт текущими skills;
   - `C:\Users\HONOR\.codex\rules\default.rules` — always-on предпочтение пользователя;
   - `AGENTS.md` — правила этого репозитория: архитектура, команды, данные, деплой;
   - `research/*.md` — локальные заметки и протоколы проекта.
5. Если меняешь skill, используй `skill-creator` и валидируй skill, когда правка существенная.
6. Делай короткие, проверяемые правки. Не дублируй одно правило сразу в skill, global rules и `AGENTS.md`.

## Проектные проверки

- Для frontend-архитектуры сначала смотри точный референс, который назвал пользователь, а потом переносишь правило в этот React/Next-проект.
- Vue-термин `composables` не копируй в React-код: файлы `use...` здесь живут в `hooks`.
- После структурных правок проверяй, что в `frontend/src` не появились старые папки `model`, `lib`, `config`, `composables`.
- Для существенных frontend-правок запускай `cd frontend && bun run build`, если пользователь не остановил проверку.
- При быстрых визуальных итерациях над цветом, шрифтом или близкими токенами не запускай сборку после каждого микрошга; проверяй diff/рендер и запускай build на устойчивой точке или по просьбе пользователя.
- Перед Playwright-проверкой существующего Next dev-сервера сначала используй уже запущенный URL. Если `bun run dev` сообщает про другой активный dev-сервер, ориентируйся на его `Local` URL/PID, а не на fallback-порт, пока новый процесс не подтвердил готовность.
- В React-хуках, где восстановление состояния откладывается через `requestAnimationFrame`/таймер, не записывай persisted-состояние в cleanup до явного флага инициализации: dev Strict Mode может вызвать ранний cleanup и затереть сохранённое значение.

## Финальный ответ

Если правила изменились, коротко скажи, какой слой обновлён и зачем. Если менять нечего, скажи, какие файлы проверены и почему новая правка не нужна.
